Behind the Scenes of Building a Multi-Service Failure Simulation Platform to Understand System Reliability

When we open an app like Uber, Netflix, or Amazon, everything appears to work instantly.

  • You request a ride.

  • You stream a movie.
  • You place an order.

But behind the scenes, these platforms are not a single application. They are made of many smaller services working together.

And those services fail all the time.

So the real question engineers face is not:

“How do we prevent failure?”

Instead, the question becomes:

“How do we design systems that continue working even when parts of them fail?”

To better understand this idea, I built a small learning project called a Multi-Service Failure Simulation Platform.

The goal of the project was simple: simulate failures in a distributed system and observe how the system reacts.

What Is a “Service” in Simple Terms?

Before diving into the project, it helps to understand what a service actually is.

A service is simply a small program responsible for one specific task.

For example, in a ride-sharing app like Uber, there might be separate services for:

  • ride requests
  • pricing calculations
  • payments
  • notifications

Instead of one large program doing everything, each service handles its own responsibility.

This architecture is called microservices, and it allows large applications to scale and evolve more easily.

However, it also introduces a challenge:

If one service fails, the others might also fail.

The Architecture of the Project

The project simulates a simplified distributed system made of several services that call each other.

The request flow looks like this:

Client → Core API → Service A → Service B → Chaos Service

Each component represents a layer in a real distributed system.

Core API: The System Entry Point

The Core API is the entry point of the system.

It represents the backend service that first receives a request from a user.

In a real application like Uber, this might be similar to the Ride Service, which receives ride requests from users.

When a request arrives, the Core API calls another service called Service A.

But instead of making a simple request, the call is protected with two important resilience mechanisms:

Retry
If a request fails, the system automatically tries again.

Circuit Breaker
If failures continue, the system temporarily stops calling the failing service to avoid overloading it.

These techniques help systems stay stable even when dependencies become unreliable.

Service A: The Business Logic Layer

Service A represents a business logic service, such as a payment or pricing service.

Its job is to process the request and call another service called Service B.

Just like the Core API, this service also uses retries and circuit breakers.

Why?

Because in distributed systems, failures often happen in downstream dependencies.

For example, imagine Uber trying to charge a credit card:

  • the first request might fail
  • the system retries automatically
  • if the bank keeps failing, the system stops calling it

This prevents a failing service from bringing down the entire system.

Service B: Simulating an External Dependency

Service B represents a dependency that the system relies on but does not control.

In real life, this could be something like:

  • a bank API
  • a payment gateway
  • a third-party service

Because these systems are external, they can fail unpredictably.

In this project, Service B calls another service called the Chaos Service, which is responsible for injecting failures.

Chaos Service: Simulating Real Production Failures

The Chaos Service intentionally introduces failures into the system.

This allows the system to be tested under different conditions.

For example:

  • 0% failure
    Everything works normally.

  • 30% failure
    Some requests fail while others succeed.

  • 100% failure
    The dependency is completely unavailable.

These scenarios mimic real-world conditions where systems may be partially degraded or fully down.

By injecting failures, we can observe how the system responds.

Observability: Seeing What Happens Inside the System

To understand how the system behaves during failures, I integrated Prometheus and Grafana.

Prometheus collects metrics such as:

  • number of retries
  • request rates
  • circuit breaker states

Grafana then visualizes those metrics in dashboards.

This makes it possible to see patterns such as retry spikes or circuit breakers opening when a service becomes unreliable.

In real engineering teams, dashboards like these are critical for diagnosing production incidents.

Simulating Traffic with Load Testing

To make the system behave more realistically, I used a tool called k6 to simulate multiple requests being sent to the system.

This helps demonstrate how the system behaves under real traffic instead of a single request.

When failures are injected, you can observe how retries increase and how the circuit breaker eventually protects the system from further calls.

What Is there to Learn from This Project?

Building this project helps to better understand how distributed systems maintain reliability.

Some key lessons include:

Failures are inevitable in distributed systems.

Retries can help recover from temporary issues but must be used carefully.

Circuit breakers prevent failing services from causing cascading outages.

Observability tools are essential for understanding system behavior.

Final Thoughts

This project is a simplified version of how real platforms maintain reliability.

Companies like Uber, Netflix, and Amazon operate systems with hundreds or thousands of services. The principles used in this small project are the same ideas that help those systems stay stable at massive scale.

Instead of trying to eliminate failure entirely, modern distributed systems are designed to detect failures quickly, isolate them, and recover automatically.

That’s the concept this project was built to explore.
